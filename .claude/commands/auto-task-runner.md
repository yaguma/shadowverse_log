# auto-task-runner

## 目的

複数のGitHub IssueまたはKairoタスクを順番に自動実行し、実装→PR→レビュー→修正のサイクルを自動化するオーケストレーションコマンドなのだ。

---

## ⚠️ 重要: 実行時チェックリスト

**各タスクで以下のチェックリストを順番に実行すること。スキップ禁止。**

```
□ Step 0: 初期化・タスク情報取得
□ Step 1: 依存関係確認
□ Step 2: 作業ブランチ作成 (/start-task)
□ Step 3: タスク実装 (/kairo-implement)
□ Step 4: PR作成 (/commit-push-pr)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ PR作成後、必ず以下を実行すること（--skip-review時を除く）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
□ Step 5: コードレビュー (/tdd-code-review) → PRコメント投稿
□ Step 6: セキュリティレビュー (/security-review) → PRコメント投稿
□ Step 7: レビュー指摘修正 (指摘がある場合のみ)
□ Step 8: 修正コミット＆プッシュ (修正した場合のみ)
□ Step 9: レビューループ判定
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔀 自動マージ（--auto-merge時のみ）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
□ Step 10: PRマージ (/pr-merge {PR番号} --strategy {戦略})
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
□ Step 11: タスク完了報告 ← ここで初めて「完了」を報告
```

**🚫 禁止事項**:
- PR作成直後に「タスク完了」を報告してはいけない
- レビューステップをスキップしてはいけない（--skip-review指定時を除く）

---

## 引数

| 引数 | 必須 | 説明 | 例 |
|------|------|------|-----|
| `requirement_name` | ○ | 要件名（docs/tasks/配下のディレクトリ名） | `atelier-guild-rank` |
| `start_id` | ○ | 開始ID（Issue番号またはタスク番号） | `#123` or `TASK-0055` |
| `--issue-mode` | - | GitHub Issue モードで実行（デフォルトはタスクモード） | - |
| `--dry-run` | - | 実行予定のタスクを表示のみ（実際には実行しない） | - |
| `--skip-review` | - | レビュー工程をスキップ（高速イテレーション用） | - |
| `--max-tasks N` | - | 最大N個のタスクを実行して停止 | `--max-tasks 3` |
| `--auto-merge` | - | PRを自動マージ（/pr-mergeを呼び出し） | - |
| `--merge-strategy` | - | マージ戦略（squash/merge/rebase）デフォルト: merge | `--merge-strategy squash` |

## 詳細手順

### Step 0: 初期化・タスク情報取得

**実行内容**:
1. 引数を解析（要件名、開始ID、オプション）
2. 開始IDの形式判定
   - `#xxx` → GitHub Issue モード
   - `TASK-xxxx` → Kairoタスクモード
3. タスクリストを構築
   - Issue: `gh issue list --state open` でオープンIssue一覧取得
   - タスク: `docs/tasks/{要件名}/overview.md` から未完了タスク抽出
4. 開始位置から順番にタスクをキューイング

**音声通知（必須）**: 「タスク自動実行を開始するのだ」

**完了条件**: タスクリストが構築されている

---

### Step 1: 依存関係確認

**実行内容**:
1. `docs/tasks/{要件名}/overview.md` から依存関係を読み込み
2. 依存タスクの完了状態を確認（ステータスが `[x]` または `completed`）
3. 未完了の依存があれば警告表示＆スキップ

**出力例**:
```
⚠️ タスク TASK-0055 は依存タスク TASK-0054 が未完了のためスキップするのだ
```

**完了条件**: 依存関係が確認され、実行可能と判断されている

---

### Step 2: 作業ブランチ作成 (`/start-task`)

**実行内容**:
1. mainブランチを最新化
2. 作業ブランチを作成（`claude/{task-id}-{short-desc}`）
3. ブランチをチェックアウト

**音声通知（必須）**: 「タスク{ID}を開始するのだ」

**完了条件**: 作業ブランチにチェックアウトしている

---

### Step 3: タスク実装 (`/kairo-implement` or `/tsumiki:kairo-implement`)

**タスク種別判定**:
1. tsumiki スキルが利用可能か確認
2. 利用可能 → `/tsumiki:kairo-implement` を使用（優先）
3. 利用不可 → `/kairo-implement` を使用（フォールバック）

**実行内容**:
1. TDDまたは直接作業プロセスで実装
2. テスト実行＆パス確認
3. 実装完了レポート生成

**音声通知（必須）**: 「実装が完了したのだ」

**完了条件**: 全テストがパスしている

---

### Step 4: PR作成 (`/commit-commands:commit-push-pr`)

**実行内容**:
1. 変更をステージング
2. コミット（Conventional Commits形式）
3. リモートにプッシュ
4. PRを作成

**出力**: PR URL（後続ステップで必ず使用）

**⚠️ 注意**: **PR作成後、Step 5に進むこと。ここで完了報告してはいけない。**

**完了条件**: PRが作成され、PR番号/URLが取得できている

---

### Step 5: コードレビュー (`/tdd-code-review`) 【必須】

**⚠️ `--skip-review` オプションが指定されていない場合、必ず実行すること**

**実行内容**:
1. 7つの観点でレビュー
   - セキュリティ
   - パフォーマンス
   - SOLID原則
   - エラーハンドリング
   - テスト品質
   - コード可読性
   - 日本語コメント品質
2. **指摘事項をPRコメントに投稿**

**音声通知（必須）**: 「レビューを開始するのだ」

**PRコメント形式**:
```bash
gh pr comment {PR番号} --body "## 🔍 コードレビュー結果

### サマリー
- Critical: {N}件
- Warning: {N}件
- Info: {N}件

### 指摘事項
{指摘内容をMarkdown形式で記載}
"
```

**完了条件**: PRにコードレビュー結果がコメントされている

---

### Step 6: セキュリティレビュー (`/security-review`) 【必須】

**⚠️ `--skip-review` オプションが指定されていない場合、必ず実行すること**

**実行内容**:
1. セキュリティ脆弱性の検出
2. **指摘事項をPRコメントに投稿**

**PRコメント形式**:
```bash
gh pr comment {PR番号} --body "## 🔒 セキュリティレビュー結果

### 検出された脆弱性
{脆弱性をMarkdown形式で記載}

### 推奨対策
{対策をMarkdown形式で記載}
"
```

**完了条件**: PRにセキュリティレビュー結果がコメントされている

---

### Step 7: レビュー指摘修正 (`/tdd-code-review-fix`)

**実行条件**: Step 5, 6でCriticalまたはWarningの指摘がある場合

**実行内容**:
1. PRコメントから指摘内容を取得（`gh pr view --comments`）
2. 優先度順に修正を実施
3. 修正完了レポート生成

**音声通知（必須）**: 「{N}件の指摘があるのだ」

**完了条件**: 指摘事項が修正されている

---

### Step 8: 修正コミット＆プッシュ

**実行条件**: Step 7で修正を行った場合

**実行内容**:
1. 修正内容をコミット（`fix: address review comments`）
2. プッシュ

**音声通知（必須）**: 「修正が完了したのだ」

**完了条件**: 修正がプッシュされている

---

### Step 9: レビューループ判定

**判定基準**:
- Critical問題: 全て修正完了（必須）
- Warning問題: 全て「修正完了」または「Issue作成済み」
- Info問題: 全て「修正完了」または「Issue作成済み」または「スキップ」

**ループ条件**: 上記を満たさない場合、Step 5に戻る（最大5回）

**出力例（完了時）**:
```
✅ レビュー完了判定:
- Critical問題: {N}件中{N}件修正完了
- Warning問題: {N}件中{M}件修正完了、{K}件Issue作成
- Info問題: {N}件中{M}件修正完了、{K}件Issue作成、{L}件スキップ
→ 全ての問題が対応済みのため、レビューループを終了
```

**出力例（タイムアウト時）**:
```
⚠️ レビューループが5回に達したのだ。手動確認が必要なのだ。
```

**完了条件**: 全ての問題が対応済み、またはループ上限到達

---

### Step 10: PRマージ (`/pr-merge`) 【オプション】

**実行条件**: `--auto-merge` オプションが指定されている場合のみ

**実行内容**:
1. `/pr-merge` スキルを呼び出し
2. CIチェック確認（デフォルト）
3. マージ戦略に応じたマージ実行（デフォルト: merge）
4. ブランチ削除（デフォルト）

**コマンド**:
```bash
# 基本呼び出し
/pr-merge {PR番号}

# 戦略指定時（--merge-strategy オプション指定時）
/pr-merge {PR番号} --strategy {squash|merge|rebase}
```

**音声通知**:
- 開始時: 「PRをマージするのだ」
- 完了時: 「マージが完了したのだ」
- 失敗時: 「マージに失敗したのだ、次のタスクに進むのだ」

**マージ失敗時の動作**:
- 警告表示して次タスクへ進む（処理を中断しない）
- 失敗したPRは後でまとめて表示

**完了条件**: PRがマージされている、またはマージ失敗が記録されている

---

### Step 11: タスク完了報告 【ここで初めて完了】

**⚠️ このステップに到達して初めて「タスク完了」を報告すること**

**実行内容**:
1. 完了サマリーを出力
2. 次のタスクがあれば Step 1 に戻る

**音声通知（必須）**: 「タスク{ID}が完了したのだ」

**出力形式**:
```
✅ タスク {TASK-ID} が完了したのだ

📊 実行サマリー:
- 実装タイプ: TDDプロセス
- PR: #{PR番号} ({PR URL})
- レビューループ: {N}回
- 指摘修正: {N}件
- マージ: ✅ 完了 ({strategy}) / ⚠️ 失敗 / - スキップ

📝 次のタスク: {次のタスクID}
```

---

```
🎉 全てのタスクが完了したのだ！

📊 実行結果サマリー:
- 実行タスク数: {N}個
- 成功: {N}個
- スキップ: {N}個（依存未完了）
- 作成PR数: {N}個
- マージ済みPR: {N}個
- マージ失敗PR: {N}個

📝 作成されたPR一覧:
- #{PR番号}: {タスクID} - {タスク名} [✅ Merged]

- #{PR番号}: {タスクID} - {タスク名} [- Not Merged]
```

**音声通知（必須）**: 「全てのタスクが完了したのだ」

## 音声通知一覧（必須）

| タイミング | メッセージ | 実行タイミング |
|------------|------------|----------------|
| コマンド開始 | 「タスク自動実行を開始するのだ」 | Step 0完了時 |
| タスク開始 | 「タスク{ID}を開始するのだ」 | Step 2完了時 |
| 実装完了 | 「実装が完了したのだ」 | Step 3完了時 |
| レビュー開始 | 「レビューを開始するのだ」 | Step 5開始時 |
| レビュー指摘あり | 「{N}件の指摘があるのだ」 | Step 7開始時 |
| 修正完了 | 「修正が完了したのだ」 | Step 8完了時 |
| マージ開始 | 「PRをマージするのだ」 | Step 10開始時（--auto-merge時） |
| マージ完了 | 「マージが完了したのだ」 | Step 10完了時（成功時） |
| マージ失敗 | 「マージに失敗したのだ、次のタスクに進むのだ」 | Step 10完了時（失敗時） |
| タスク完了 | 「タスク{ID}が完了したのだ」 | Step 11完了時 |
| 全タスク完了 | 「全てのタスクが完了したのだ」 | 全タスク終了時 |
| エラー発生 | 「エラーが発生したのだ」 | エラー発生時 |

**音声通知設定**: `speaker=3, speedScale=1.3`

---

## エラーハンドリング

| エラー | 対応 |
|--------|------|
| 引数不足 | エラーメッセージを表示し、使用方法を案内 |
| 要件名が存在しない | `docs/tasks/{要件名}` が見つからない場合はエラー終了 |
| タスクIDが存在しない | 該当タスクが見つからない場合はスキップして次へ |
| 依存タスク未完了 | 警告表示してスキップ、次のタスクへ |
| 実装エラー | エラー詳細を表示し、該当タスクで停止 |
| レビューループ上限 | 5回到達で警告表示し、手動確認を促す |
| PRコメント投稿失敗 | リトライ（最大3回）後、ローカルにログ出力 |

## 使用例

```bash
# タスクモードで実行
/auto-task-runner atelier-guild-rank TASK-0055

# Issueモードで実行
/auto-task-runner atelier-guild-rank #123 --issue-mode

# ドライラン（実行計画のみ表示）
/auto-task-runner atelier-guild-rank TASK-0055 --dry-run

# レビューをスキップして高速実行
/auto-task-runner atelier-guild-rank TASK-0055 --skip-review

# 3タスクまで実行して停止
/auto-task-runner atelier-guild-rank TASK-0055 --max-tasks 3

# 自動マージ付きで実行（デフォルト: mergeマージ）
/auto-task-runner atelier-guild-rank TASK-0055 --auto-merge

# マージ戦略を指定（squash）
/auto-task-runner atelier-guild-rank TASK-0055 --auto-merge --merge-strategy squash

# レビュースキップ + 自動マージ（高速イテレーション用）
/auto-task-runner atelier-guild-rank TASK-0055 --skip-review --auto-merge

# 複合オプション
/auto-task-runner atelier-guild-rank TASK-0055 --max-tasks 5 --skip-review --auto-merge --merge-strategy squash
```

---

## 注意事項

1. **PR作成後に必ずレビューを実行すること**（最重要）
2. **タスク完了報告はStep 10でのみ行うこと**
3. 各サブエージェントは `general-purpose` タイプで実行するのだ
4. VOICEVOXによる音声通知を各ステップで必ず実施するのだ
5. レビューループは最大5回まで（無限ループ防止）なのだ
6. 各ステップでエラーが発生した場合は即座に停止し、状態を報告するのだ
7. `--dry-run` オプションで実行前に計画を確認できるのだ
